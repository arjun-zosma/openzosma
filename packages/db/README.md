# @openzosma/db

Database package for OpenZosma. Contains all migrations, connection pooling, type definitions, and query modules.

## Overview

OpenZosma uses **PostgreSQL** with two schemas:

- **`public`** -- Application tables (gateway + web app). Managed by [`db-migrate`](https://db-migrate.readthedocs.io/).
- **`auth`** -- Authentication tables (users, sessions, accounts, verifications). Managed by the [`better-auth` CLI](https://www.better-auth.com/).

There is **no ORM**. All queries use raw SQL via the [`pg`](https://node-postgres.com/) driver.

## Prerequisites

A running PostgreSQL instance with the [pgvector](https://github.com/pgvector/pgvector) extension available. The default Docker Compose setup (`docker compose up -d` from the repo root) uses the `pgvector/pgvector:pg16` image, which includes pgvector. Default connection:

- **Database:** `openzosma`
- **User/Password:** `openzosma` / `openzosma`
- **Host/Port:** `localhost:5432`

If you manage PostgreSQL yourself, install the pgvector extension before running migrations (the knowledge base migration requires it).

### ESM Compatibility

The `packages/db` package uses `"type": "module"` in its `package.json`, but `db-migrate` expects CommonJS migration files. A `migrations/package.json` with `"type": "commonjs"` overrides this so that the CJS migration boilerplate files (generated by `db-migrate create --sql-file`) work correctly. Do not remove this file.

## Environment Variables

Migration scripts load env files automatically. They check for `.env.local` first, then `.env` in the repo root. You can override this with `--env-file=<path>`.

| Variable | Default | Description |
|---|---|---|
| `DATABASE_URL` | _(built from DB\_\* vars)_ | Full PostgreSQL connection string. Takes precedence if set. |
| `DB_HOST` | `localhost` | PostgreSQL host |
| `DB_PORT` | `5432` | PostgreSQL port |
| `DB_NAME` | `openzosma` | Database name |
| `DB_USER` | `openzosma` | Database user |
| `DB_PASS` | `openzosma` | Database password |
| `DB_POOL_SIZE` | `20` | Connection pool size (used by `createPool()`, not migrations) |

If `DATABASE_URL` is not set, it is constructed as `postgresql://<DB_USER>:<DB_PASS>@<DB_HOST>:<DB_PORT>/<DB_NAME>`.

## Migrations

### Quick Start

From the repo root:

```bash
# 1. Run public schema migrations (gateway + web app tables)
pnpm db:migrate

# 2. Run auth schema migrations (better-auth tables in auth schema)
pnpm db:migrate:auth
```

**Order matters.** Run `db:migrate` first, then `db:migrate:auth`.

Or from this package directory:

```bash
pnpm migrate
pnpm migrate:auth
```

### How It Works

#### Public Schema Migrations (`db-migrate`)

`src/migrate.ts` is the entrypoint. It:

1. Loads environment variables (`.env.local` -> `.env`, or explicit `--env-file=<path>`)
2. Constructs `DATABASE_URL` from `DB_*` vars if not already set
3. Shells out to `npx db-migrate up --env production --migrations-dir migrations`

The `database.json` config file tells db-migrate how to connect:

- `dev` environment: hardcoded local connection string
- `production` environment: reads `DATABASE_URL` from env (used by `migrate.ts`)

Each migration consists of three files:

```
migrations/
  <timestamp>-<name>.js           # JS boilerplate that reads the SQL files
  sqls/<timestamp>-<name>-up.sql   # SQL to apply the migration
  sqls/<timestamp>-<name>-down.sql # SQL to reverse the migration
```

The JS file is standard db-migrate boilerplate -- it reads the corresponding SQL file and executes it. All migration logic lives in the SQL files.

#### Auth Schema Migrations (`better-auth` CLI)

`src/migrate-auth.ts` is the entrypoint. It:

1. Loads environment variables (same logic as `migrate.ts`)
2. Connects to PostgreSQL and creates the `auth` schema if it doesn't exist (`CREATE SCHEMA "auth"`)
3. Shells out to `pnpx @better-auth/cli migrate --yes` with `cwd` set to `apps/web/` (where the `auth.ts` config file lives)

The better-auth CLI reads the auth configuration from `apps/web/src/lib/auth.ts` and generates/runs the necessary migrations. These SQL files are stored in `better-auth_migrations/` for reference but are managed by the CLI.

### Rolling Back

```bash
# From repo root
pnpm db:migrate:down

# From this package
pnpm migrate:down
```

This rolls back the last public schema migration. Auth migrations do not have a built-in rollback.

### Creating New Migrations

```bash
# From repo root
pnpm db:migrate:create -- <name>

# From this package
pnpm migrate:create -- <name>
```

This generates the JS boilerplate and empty `-up.sql` / `-down.sql` files in `migrations/sqls/`. Write your SQL in those files.

### Using an Explicit Env File

```bash
pnpm migrate -- --env-file=/path/to/.env.production
pnpm migrate:auth -- --env-file=/path/to/.env.production
```

### Naming Conventions

Gateway tables use `snake_case` column names (`created_at`, `user_id`). Web app tables use `flatcase` column names (`createdat`, `organizationid`) with `public.` schema prefix in SQL. Both coexist in the `public` schema.

## Query Modules

This package exports typed query functions for gateway tables:

```typescript
import {
  createPool,
  agentConfigQueries,
  apiKeyQueries,
  usageQueries,
  connectionQueries,
  settingQueries,
  userSandboxQueries,
} from "@openzosma/db"

const pool = createPool()

// Example: get all agent configs
const configs = await agentConfigQueries.getAll(pool)

// Example: record usage
await usageQueries.recordUsage(pool, { sessionId: "...", tokens: 100 })

// Example: get a user's sandbox
const sandbox = await userSandboxQueries.getByUserId(pool, "user-id")
```

### Exported Types

- `AgentConfig` -- Agent configuration record
- `ApiKey` -- API key record
- `UsageRecord` -- Usage tracking record
- `Connection` -- External connection record
- `ConnectionType` -- Union type for connection types
- `Setting` -- Key-value setting record
- `UserSandbox` -- Per-user sandbox record (id, user_id, sandbox_name, status, metadata, timestamps)
- `SandboxStatus` -- Sandbox lifecycle status (`creating` | `ready` | `suspended` | `failed` | `destroyed`)
- `PoolConfig` -- Configuration for `createPool()`

## File Structure

```
packages/db/
  database.json                         # db-migrate connection config
  package.json
  tsconfig.json
  src/
    index.ts                            # Public API (re-exports)
    pool.ts                             # createPool() using pg.Pool
    types.ts                            # TypeScript interfaces
    migrate.ts                          # CLI: public schema migrations
    migrate-auth.ts                     # CLI: auth schema migrations
    queries/
      agent-configs.ts                  # CRUD for agent_configs
      api-keys.ts                       # CRUD for api_keys
      usage.ts                          # Usage tracking queries
      connections.ts                    # CRUD for connections
      settings.ts                       # CRUD for settings
      user-sandboxes.ts                 # CRUD for user_sandboxes
  migrations/
    package.json                        # "type": "commonjs" override for db-migrate CJS files
    <timestamp>-<name>.js               # db-migrate JS boilerplate files
    sqls/
      <timestamp>-<name>-up.sql         # UP migration SQL
      <timestamp>-<name>-down.sql       # DOWN migration SQL
  better-auth_migrations/
    <timestamp>.sql                     # better-auth CLI managed SQL files
```
